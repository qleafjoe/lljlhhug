## 一、性能测试基础概念

### 1.1 什么是性能测试？
性能测试是通过**模拟生产环境**的负载场景，验证系统在**不同压力级别**下的性能表现，确保系统满足预期的性能指标。

### 1.2 为什么要做性能测试？
- 验证系统能否承载预期用户量
- 发现系统性能瓶颈和潜在问题
- 评估系统的稳定性和可靠性
- 为系统容量规划提供数据支撑
- 优化资源配置，降低运营成本

### 1.3 性能测试的目标
| 目标 | 说明 |
|------|------|
| 响应时间 | 请求到响应的耗时 |
| 吞吐量 | 单位时间处理的请求数（TPS/QPS） |
| 资源利用率 | CPU、内存、磁盘IO、网络等 |
| 稳定性 | 长时间运行是否正常 |
| 并发能力 | 支持多少用户同时访问 |

## 二、性能测试类型

### 2.1 基准测试（Benchmark Test）
- 定义：在标准条件下进行的测试，作为性能参考基准
- 用途：新系统上线前的基线对比
- 特点：单用户、单请求

### 2.2 负载测试（Load Test）
- 定义：模拟真实用户负载，测试系统在预期负载下的性能
- 用途：验证系统在正常负载下的表现
- 特点：逐渐增加负载，达到预期峰值

### 2.3 压力测试（Stress Test）
- 定义：超过系统设计能力，测试系统的极限和故障恢复能力
- 目的：找出系统的崩溃点，验证容错能力
- 特点：持续增加压力直到系统失效

### 2.4 稳定性测试（ Endurance / Soak Test）
- 定义：长时间（数小时/数天）在一定负载下运行
- 目的：检测内存泄漏、连接池耗尽等长时间运行问题
- 特点：负载一般为预期的70%-80%

### 2.5 并发测试（Concurrency Test）
- 定义：多个用户同时访问同一功能，测试并发处理能力
- 目的：发现线程安全、资源竞争问题
- 特点：同一时刻发起请求

### 2.6 峰值测试（Spike Test）
- 定义：短时间内突然大幅增加负载
- 目的：验证系统应对流量突增的能力
- 特点：负载急剧上升然后下降

### 各类测试对比

| 测试类型 | 负载特征 | 测试目的 | 持续时间 |
|----------|----------|----------|----------|
| 基准测试 | 最小负载 | 建立基线 | 短 |
| 负载测试 | 预期负载 | 验证正常性能 | 中 |
| 压力测试 | 超负荷 | 找极限 | 短 |
| 稳定性测试 | 70-80%预期 | 检测内存泄漏 | 长 |
| 并发测试 | 同时请求 | 验证并发能力 | 短 |
| 峰值测试 | 突增突减 | 应对峰值能力 | 中 |

## 三、核心性能指标

### 3.1 响应时间（Response Time）

| 指标 | 说明 | 公式 |
|------|------|------|
| 平均响应时间 | 所有请求耗时的平均值 | 总时间/请求数 |
| 90%响应时间 | 90%的请求在此时间内完成 | 百分位数 |
| 95%响应时间 | 95%的请求在此时间内完成 | 百分位数 |
| 99%响应时间 | 99%的请求在此时间内完成 | 百分位数 |
| 最大响应时间 | 最慢请求的响应时间 | - |

> 💡 面试重点：为什么用90%/95%而不是平均值？
> - 平均值容易被异常值拉高/拉低，不能反映大部分用户的真实体验
> - 百分位数更能反映用户体验的真实性

### 3.2 吞吐量（Throughput）

| 指标 | 说明 | 单位 |
|------|------|------|
| TPS | Transactions Per Second，每秒事务数 | transactions/sec |
| QPS | Queries Per Second，每秒查询数 | queries/sec |
| RPS | Requests Per Second，每秒请求数 | requests/sec |

> 💡 TPS vs QPS 区别：
> - TPS：一个事务可能包含多个请求
> - QPS：每秒查询数，通常指单机
> - 大厂常问：1个TPS等于多少QPS？→ 取决于事务包含的请求数

### 3.3 并发数（Concurrency）
- **虚拟用户数（VU）**：模拟的用户数量
- **并发线程数**：同时发起请求的线程数
- **在线用户数**：实际登录系统的用户数

### 3.4 资源利用率

| 资源 | 正常范围 | 说明 |
|------|----------|------|
| CPU | < 70% | 持续 > 80% 表示CPU瓶颈 |
| 内存 | < 80% | 持续增长可能内存泄漏 |
| 磁盘IO | < 70% | IO等待影响响应时间 |
| 网络 | < 50% | 带宽是否成为瓶颈 |

### 3.5 错误率
- **失败率**：请求失败的比例
- **超时率**：请求超时的比例
- 合格标准：一般 < 1%，核心接口 < 0.1%

## 四、性能测试流程

### 4.1 性能测试生命周期

```
┌─────────────────────────────────────────────────────────────┐
│  需求分析 → 计划制定 → 场景设计 → 环境搭建 → 脚本开发        │
│      ↓                                                    │
│  执行监控 ← 数据分析 ← 性能调优 ← 问题定位 ← 报告输出        │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 详细流程步骤

#### Step 1: 需求分析
- 明确性能测试目标
- 确定性能指标阈值
- 分析业务场景和用户行为
- 确定测试范围

#### Step 2: 制定测试计划
- 测试策略选择
- 人员分工安排
- 时间计划
- 风险评估

#### Step 3: 设计测试场景
- 业务模型设计
- 虚拟用户脚本设计
- 数据准备
- 场景配置

#### Step 4: 搭建测试环境
- 硬件配置（与生产一致或等比）
- 软件环境配置
- 网络配置
- 监控工具部署

#### Step 5: 开发测试脚本
- 录制/编写脚本
- 参数化配置
- 关联处理
- 事务和检查点

#### Step 6: 执行测试
- 梯度加压
- 实时监控
- 数据收集
- 记录日志

#### Step 7: 分析结果
- 性能数据分析
- 瓶颈定位
- 对比基线
- 给出结论

#### Step 8: 性能调优
- 代码优化
- 数据库优化
- 服务器优化
- 架构优化

#### Step 9: 输出报告
- 测试概要
- 性能数据
- 问题清单
- 优化建议

## 五、性能测试工具

### 5.1 主流工具对比

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| JMeter | 开源、功能强大、插件丰富 | Web、移动端、接口测试 |
| LoadRunner | 商业、功能全面、支持广泛 | 企业级大型项目 |
| Locust | Python编写、可扩展、分布式 | 喜欢Python的团队 |
| Gatling | Scala编写、报告美观 | 现代Web应用 |
| ab (Apache Bench) | 简单易用、单机压测 | 快速简单测试 |

### 5.2 JMeter 详解（重点！）

#### 5.2.1 JMeter核心组件
```
测试计划 (Test Plan)
  ├── 线程组 (Thread Group)      - 用户模拟配置
  ├── 配置元件 (Config Elements) - 参数化、CSV数据
  ├── 前置处理器 (Pre Processors) - 请求前处理
  ├── 定时器 (Timers)            - 思考时间
  ├── 取样器 (Samplers)          - 发起请求
  ├── 后置处理器 (Post Processors) - 响应数据提取
  ├── 断言 (Assertions)          - 结果验证
  └── 监听器 (Listeners)         - 结果展示
```

#### 5.2.2 线程组配置
- **线程数**：模拟的用户数（并发数）
- **Ramp-up时间**：启动所有线程的时间（避免瞬间高压）
- **循环次数**：每个线程执行的次数
- **调度器**：设置持续时间和启动延迟

```
线程数: 100
Ramp-up: 10秒
循环次数: 10
= 每秒启动 10 个用户，共 1000 次请求
```

#### 5.2.3 JMeter常用Sampler
| Sampler | 说明 |
|---------|------|
| HTTP Request | HTTP接口请求 |
| JDBC Request | 数据库操作 |
| TCP Sampler | TCP协议请求 |
| JMS Producer | 消息队列 |
| OS Process | 启动外部程序 |

#### 5.2.4 JMeter参数化方式
1. **用户定义的变量** - 固定值
2. **CSV Data Set Config** - 从文件读取
3. **随机函数** - ${__Random(1,100)}
4. **计数器** - ${__counter(TRUE,)}

#### 5.2.5 关联处理
- **正则表达式提取器**：提取响应中的动态数据
- **JSON提取器**：提取JSON响应中的数据
- **XPath提取器**：提取XML响应中的数据

#### 5.2.6 常用函数
```
${__time()}                    // 当前时间戳
${__time(yyyy-MM-dd)}         // 格式化时间
${__Random(1,100)}           // 随机数
${__CSVRead(file,col)}        // 读取CSV
${__property(prop)}           // 读取属性
${__P(prop,default)}         // 读取参数
```

### 5.3 JMeter 高级特性

#### 5.3.1 分布式压测
- **Master**：控制机，负责分发任务
- **Slave**：执行机，负责发起请求
- 配置：`jmeter.properties` 中设置 `remote_hosts`

#### 5.3.2 调试Sampler
- **Debug Sampler**：输出变量信息
- **View Results Tree**：查看请求/响应详情

#### 5.3.3 报告生成
```bash
# 生成HTML报告
jmeter -g result.jtl -o ./report

# 命令行执行
jmeter -n -t test.jmx -l result.jtl -e -o ./report
```

## 六、常见性能问题分析

### 6.1 响应时间长

#### 原因分析
```
响应时间长
├── 网络问题
│   ├── 网络带宽不足
│   ├── 网络延迟高
│   └── DNS解析慢
├── 服务器问题
│   ├── CPU利用率高
│   ├── 内存不足/泄漏
│   ├── 磁盘IO瓶颈
│   └── 连接池耗尽
├── 数据库问题
│   ├── 缺少索引
│   ├── SQL性能差
│   ├── 连接数不足
│   └── 数据库锁竞争
└── 代码问题
    ├── 循环/递归效率低
    ├── 同步阻塞
    └── 算法复杂度高
```

#### 定位方法
1. **分阶段耗时分析**：前端 → DNS → 连接 → 服务器 → 数据库
2. **APM工具**：SkyWalking、Pinpoint、Cat
3. **监控工具**：Prometheus + Grafana

### 6.2 吞吐量低

#### 常见原因
- CPU成为瓶颈
- 数据库连接池配置过小
- 线程池配置不当
- 同步阻塞操作
- 网络IO阻塞
- 磁盘IO瓶颈

### 6.3 内存问题

#### 内存泄漏
- 表现：内存持续增长，最终OOM
- 常见原因：
  - 静态集合类持有引用
  - 监听器未移除
  - 线程未结束
  - 缓存无限增长

#### 内存溢出 (OOM)
- **堆溢出**：对象创建过多
- **栈溢出**：递归调用过深
- **方法区溢出**：类加载过多

### 6.4 数据库性能问题

#### 常见问题
| 问题 | 表现 | 解决方案 |
|------|------|----------|
| 慢查询 | QPS低 | 添加索引、优化SQL |
| 连接池耗尽 | 请求等待 | 调整连接池大小 |
| 锁竞争 | 响应变慢 | 优化事务、降低隔离级别 |
| 主从延迟 | 数据不一致 | 读写分离、延迟处理 |

## 七、大厂面试常考问题

### 7.1 基础概念类

#### Q1: 什么是性能测试？和功能测试有什么区别？
**答**：
- 性能测试：验证系统在特定负载下的响应时间、吞吐量、资源利用率
- 功能测试：验证功能的正确性
- 区别：功能测试关注"能不能用"，性能测试关注"好不好用"

#### Q2: 性能测试有哪些类型？
**答**：
- 基准测试：建立性能基线
- 负载测试：验证预期负载下的性能
- 压力测试：超过设计能力测试极限
- 稳定性测试：长时间运行检测内存泄漏
- 并发测试：多用户同时访问
- 峰值测试：突发流量测试

#### Q3: TPS和QPS的区别是什么？
**答**：
- TPS：每秒事务数，一个事务可能包含多个请求
- QPS：每秒查询数，通常指单机
- 例如：1次登录 = 1次查询用户 + 2次查询权限 = 3 QPS = 1 TPS

### 7.2 工具使用类

#### Q4: JMeter有哪些组件？线程组配置参数含义？
**答**：
- 组件：线程组、采样器、控制器、定时器、断言、监听器、后置处理器
- 线程组参数：
  - 线程数：并发用户数
  - Ramp-up：启动时间（避免瞬间高压）
  - 循环次数：每个用户执行次数

#### Q5: JMeter如何做参数化？
**答**：
- CSV Data Set Config：从文件读取
- 用户定义的变量：固定值
- 函数：${__Random()}、${__time()}

#### Q6: JMeter关联怎么处理？
**答**：
- 正则表达式提取器：提取响应中的数据到变量
- JSON提取器：提取JSON数据
- XPath提取器：提取XML数据

### 7.3 场景设计类

#### Q7: 如何设计一个性能测试场景？
**答**：
1. 分析业务模型，确定主要业务流程
2. 根据业务比例分配虚拟用户
3. 确定性能指标（TPS、响应时间）
4. 设计梯度加压方案
5. 添加思考时间模拟真实用户
6. 配置监控和断言

#### Q8: 如何确定并发数？
**答**：
- 方法1：日活用户 × 峰值比例 / 日均请求数
- 方法2：根据业务预估，如同时下单用户
- 方法3：参考历史数据或行业标准

### 7.4 问题定位类

#### Q9: 响应时间很长，如何定位问题？
**答**：
1. 分阶段分析：前端→网络→应用→数据库
2. 使用APM工具定位具体耗时
3. 检查CPU、内存、磁盘、网络资源
4. 检查数据库慢查询
5. 查看应用日志和线程堆栈

#### Q10: 内存泄漏如何排查？
**答**：
1. 使用jmap导出堆内存快照
2. 使用MAT分析内存对象
3. 长时间压测观察内存趋势
4. 检查代码中的资源未关闭、缓存等问题

### 7.5 优化建议类

#### Q11: 性能测试发现问题后如何优化？
**答**：
- 代码层：优化算法、减少循环、使用缓存
- 数据库：添加索引、优化SQL、读写分离
- 服务器：增加资源、调整JVM参数
- 架构层：负载均衡、CDN、异步处理

#### Q12: 如何评估系统容量？
**答**：
1. 确定核心指标（TPS、响应时间）
2. 进行压力测试找到系统极限
3. 预留20-30%的冗余容量
4. 给出推荐配置和扩容建议

### 7.6 场景实战类

#### Q13: 如果系统响应变慢，你会怎么排查？
**答**：
1. 检查系统资源（CPU、内存、磁盘、网络）
2. 查看数据库慢查询日志
3. 使用APM工具定位具体慢请求
4. 检查是否有大批量任务在执行
5. 查看应用错误日志
6. 分析线程状态，看是否有阻塞

#### Q14: 压测过程中TPS上不去是什么原因？
**答**：
- CPU瓶颈：计算密集型任务
- 线程池/连接池配置过小
- 同步阻塞导致线程等待
- 数据库锁竞争
- 网络带宽限制
- 压测机性能不足

## 八、实战问题汇总

### 8.1 命令行问题

#### Q15: 如何在Linux查看CPU/内存/磁盘使用情况？
```bash
# CPU和内存
top
free -h

# 磁盘
df -h

# 详细CPU信息
mpstat -P ALL 1

# 磁盘IO
iostat -x 1
```

#### Q16: 如何查看端口占用和网络连接？
```bash
# 端口占用
netstat -tuln | grep 8080
ss -tuln | grep 8080

# 网络连接状态
netstat -an | grep ESTABLISHED
ss -s
```

### 8.2 JMeter实际问题

#### Q17: JMeter压测结果如何分析？
- 查看聚合报告（Aggregate Report）
- 关注：Average、90%/95% Line、Error%、Throughput
- 对比不同场景的结果
- 查看错误日志定位问题

#### Q18: JMeter分布式压测如何配置？
- Master配置slave地址
- 防火墙开放1099端口
- 确保JMeter版本一致
- 使用同一套脚本和数据
